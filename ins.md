# 目的
Throttlyのような「通信制限（ネットワーク・コンディショナー）」アプリをAndroidで作る。ネットワークの低レイヤーを学びながら作る

# 仕組みの全体像
通常、通信は「アプリ → インターネット」と直接流れますが、このアプリでは VpnService を使って通信を自分自身に引き込みます。

アプリ（Chromeなど） が通信しようとする。

Android OSが通信を TUN（仮想ネットワーク端末） に転送する。

自作アプリ がTUNからパケットを1つずつ読み出す。

ここで「わざと待つ」あるいは「捨てる」 ことで通信を絞る。

処理したパケットをインターネットへ流す。

# 🛠️ ステップ・バイ・ステップ開発フロー
Step 1: 権限とプロジェクトの準備
まずは「このアプリは通信を横取りします」という宣言をOSに対して行います。

作業: AndroidManifest.xml に BIND_VPN_SERVICE 権限を持つ Service を登録します。

学ぶこと: Androidのセキュリティモデル（VPNの許可ダイアログが出る仕組み）。

Step 2: VPNを「起動」させる
ボタンを押したら、ステータスバーに「鍵マーク」が出る状態を目指します。

作業: VpnService.Builder を使い、仮想のIPアドレス（例: 10.0.0.1）とルートを設定して establish() を呼び出します。

学ぶこと: * TUNインターフェース: 物理的なLANカードではない「仮想の口」の概念。

ルーティング: なぜ 0.0.0.0/0 を指定すると全通信が自分に来るのか。

Step 3: パケットの「読み書き」ループ（心臓部）
VPNを確立すると、ファイルのように通信データを読み書きできるようになります。

作業: ParcelFileDescriptor から FileInputStream（受信）と FileOutputStream（送信）を取得し、while ループでパケットを回します。

学ぶこと: * IPパケットの構造: データの塊がどのようなヘッダー（宛先IPなど）を持っているか。

バイナリ操作: ByteBuffer を使ったデータの扱い。

Step 4: 通信を絞る（スロットリングの実装）
ここでいよいよ「体験」の部分です。1秒間に通していいデータ量（バイト数）を決め、それを超えそうなら Thread.sleep() で処理を遅延させます。

作業: 1.  パケットのサイズを取得する。
2.  「目標速度（例: 100KB/s）」に対して、今のパケットを送るのに必要な時間を計算する。
3.  計算した時間分だけスレッドを止める。

学ぶこと: トークンバケット・アルゴリズム（通信制限の一般的なロジック）。

Step 5: 通信を「外」へ逃がす（最難関）
Step 4までだと、通信を「受け取って遅延させる」ことはできますが、それを実際のインターネットへ届ける仕組みがありません。

現実的な解決策: 最初は全てを自分で書かず、オープンソースのライブラリ（例: tun2socks）がどうやってパケットを実際の通信に変換しているか調べるのが近道です。

学ぶこと: NAT（ネットワークアドレス変換）。プライベートIP（10.0.0.1）をグローバルIPに変換して外に出す仕組み。

💡 最初にやるべき「最小体験」
いきなり全アプリの通信を絞るのは難しいので、まずは 「VPNをONにしたら、特定のアプリの通信だけが遮断される（ブラックホールVPN）」 を作ってみるのがおすすめです。

VpnService.Builder で addAllowedApplication("com.android.chrome") を指定。

while ループの中で、読み取ったパケットをどこにも書き込まずに捨てる。

Chromeだけネットに繋がらなくなれば成功です！

まずはこの「通信を遮断する（速度0）」という体験から始めてみませんか？ 準備ができたら、具体的なコード（VPNサービスの立ち上げ部分）を提示することも可能です。

# memo
Phase 1 — ブラックホールVPN（通信遮断）
Phase 2a: DNS転送（UDP port 53）— 名前解決だけ通す
Phase 2b: 汎用 UDP 転送
Phase 2c: TCP 転送（TCP ステートマシンの実装 or lwIP/netstack 等のライブラリ利用）
Phase 3: トークンバケットで速度制限

① DNS (UDP port 53) ：「住所録」アプリが api.instagram.com に繋ぎたいとき、まずはそのIPアドレスを誰かに聞かなければなりません。なぜ分ける？: これが通らないと、アプリは「相手がどこにいるか分からない」状態になり、TCPやUDPの通信を始めることすらできません。いわば**「通信の着火剤」**です。
② UDP ：「スピード重視の投げっぱなし」用途: ビデオ通話（LINE/Zoom）、オンラインゲーム、動画ストリーミング（YouTubeの一部）。特徴: 届いたかどうかを確認せず、どんどん送ります。多少データが欠けても「今の映像を早く届ける」ことが優先される通信です。
③ TCP ：「確実性重視の契約書」用途: Webサイト閲覧（HTTP/HTTPS）、SNSのタイムライン、ファイルのダウンロード、API通信。特徴: 「送ったよ」「届いたよ」「次はこれ送るね」と、一文字も間違えないように丁寧にやり取りします。**スマホの通信の主役（$80 \sim 90\%$ 以上）**はこれです。
